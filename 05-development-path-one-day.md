# 05 Development Path One Day

## 总体策略

1 天开发的关键不是做完所有功能，而是严格控制顺序。

建议原则：
1. 先定结构，再写页面
2. 先跑通主链路，再做质量提升
3. 先保证结果可展示，再做漂亮
4. 先做同步流程，再做历史和迭代

## 推荐开发节奏

### 阶段 1：第 0 到 1.5 小时

目标：
固定 MVP 范围和结果 schema。

要完成：
1. 建项目基础结构
2. 确定页面路由
3. 确定 result schema
4. 确定 API 路由清单
5. 准备环境变量和模型调用方式

交付物：
1. 能跑起来的前端壳子
2. 后端接口骨架
3. 固定的结果 JSON 样例

### 阶段 2：第 1.5 到 4 小时

目标：
打通从输入到结果的最小闭环。

要完成：
1. Landing 页面
2. Workspace 页面
3. 表单提交
4. 单图上传
5. `/api/tasks`
6. `/api/tasks/:taskId/analyze`
7. Result 页面基础渲染

交付物：
1. 用户可输入主题并提交
2. 用户可上传图片
3. 页面可显示结构化结果

### 阶段 3：第 4 到 7 小时

目标：
让结果更像真正的设计工具输出。

要完成：
1. 配色系统模块
2. 图片适配模块
3. 基础规则层
4. CSS Variables 输出
5. Tailwind 片段输出

交付物：
1. 有色板
2. 有图片适配建议
3. 有导出片段

### 阶段 4：第 7 到 9 小时

目标：
补齐最必要的质量层。

要完成：
1. 对比度提醒
2. 图片叠字风险提示
3. 页面空状态与错误状态
4. loading 状态
5. 结果页层次优化

交付物：
1. 结果可理解
2. 错误流程可控
3. 页面可演示

### 阶段 5：第 9 到 12 小时

目标：
做最后的展示和收尾。

要完成：
1. Hero 和结果页文案优化
2. Moodboard 样式增强
3. 复制按钮可用
4. 测试 2 到 3 个输入案例
5. 准备 demo 叙事

交付物：
1. 可演示 MVP
2. 稳定的示例数据
3. 演示路径清晰

## 具体开发路径

### Path A：项目骨架

1. 初始化前端项目
2. 配置样式系统
3. 创建三页路由
4. 配置基础 API 路由

### Path B：任务主链路

1. 完成输入表单
2. 接入图片上传
3. 创建任务
4. 触发分析
5. 跳转结果页

### Path C：结果渲染

1. 写死一个 mock schema
2. 让前端先基于 mock schema 跑通
3. 再接真实接口

说明：
这一步非常推荐，可以显著降低耦合和返工风险。

### Path D：规则层最小实现

1. 提取基础调色板
2. 计算对比度提醒
3. 输出 token 结构

### Path E：体验层

1. Moodboard 卡片排布
2. 复制按钮
3. loading 与错误反馈
4. 页面视觉统一

## 一天内最该砍掉的内容

如果时间不够，优先砍这些：
1. 历史记录
2. 版本管理
3. 偏好微调
4. Split View
5. 复杂动画

不要砍这些：
1. 结构化结果
2. 配色系统
3. 图片适配
4. Token 导出
5. Result 页面主链路
